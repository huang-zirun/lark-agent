# DevFlow Engine 技术方案设计文档

## 1. 项目概述

### 1.1 项目背景
DevFlow Engine 是一个基于 AI 驱动的研发流程自动化引擎，参加飞书 AI 校园挑战赛的产品创新赛道课题三："基于 AI 驱动的需求交付流程引擎"。该项目旨在将传统软件研发流程从"人力密集型流水线"转变为"AI 驱动的自动化 Pipeline"。

### 1.2 核心理念
**Pipeline 是骨架，Agent 是肌肉，人类做决策**。用户通过飞书消息或 API 输入自然语言需求，系统自动拆解为 6 阶段 AI Agent Pipeline，每个阶段由专门的 Agent 执行，人类在关键检查点进行 Approve/Reject 决策，最终产出可交付的代码变更。

### 1.3 核心价值主张
1. **需求驱动开发**：从自然语言需求直接生成可执行代码
2. **人机协同**：AI 负责执行，人类负责监督，实现高效协作
3. **全流程自动化**：覆盖需求分析、方案设计、编码、测试、评审、交付的完整闭环
4. **可观测性**：提供实时监控、过程可视化和详细审计日志

## 3. Pipeline 流程设计

### 3.1 6 阶段 Agent Pipeline

```
用户输入需求（飞书消息 / 文档 / API）
        │
        ▼
┌─────────────────┐
│  需求分析 Agent   │  理解意图，澄清歧义，输出结构化需求
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  方案设计 Agent   │  分析代码库上下文，输出技术方案
└────────┬────────┘
         │
         ▼
    ╔═══════════╗
    ║ 检查点 ①  ║  人工审批方案（飞书审批 / Bot 消息）
    ╚═════╤═════╝
          │ Approve
          ▼
┌─────────────────┐
│  代码生成 Agent   │  按方案逐文件生成/修改代码
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  测试生成 Agent   │  检测测试框架，生成/补充测试并执行
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  代码评审 Agent   │  只读审查 + 一次自动修复机会
└────────┬────────┘
         │
         ▼
    ╔═══════════╗
    ║ 检查点 ②  ║  人工确认代码评审
    ╚═════╤═════╝
          │ Approve
          ▼
┌─────────────────┐
│   交付 Agent     │  打包变更、测试证据、评审结论、合并建议
└─────────────────┘
```

### 3.2 各阶段详细设计

#### 3.2.1 需求分析 Agent
- **输入**：自然语言需求描述
- **输出**：`devflow.requirement.v1` 结构化 JSON
- **功能**：
  - LLM 分析和离线启发式分析双模式
  - 支持飞书文档、消息、Bot 事件等多种来源
  - 自动创建 PRD 飞书文档并回复交互卡片
  - 结构化用户故事、验收标准、质量信号

#### 3.2.2 方案设计 Agent
- **输入**：结构化需求 + 代码库上下文
- **输出**：`devflow.solution_design.v1` 技术方案 JSON
- **功能**：
  - AST 语义索引分析代码库结构
  - 支持已有仓库和全新项目两种模式
  - 输出变更清单、API 设计、技术选型建议

#### 3.2.3 代码生成 Agent
- **输入**：技术方案 + 代码库
- **输出**：`devflow.code_generation.v1` JSON + diff
- **功能**：
  - 受限文件工具集：read/write/edit/glob/grep/semantic_search + powershell
  - workspace 边界检查和安全审计
  - 多轮工具调用循环，最多 8 轮

#### 3.2.4 测试生成 Agent
- **输入**：代码变更集 + 需求
- **输出**：`devflow.test_generation.v1` JSON + diff
- **功能**：
  - 自动检测测试框架（pytest/unittest, npm test, Maven/Gradle）
  - 优先复用成熟框架生成/补充测试
  - 执行测试并记录结果

#### 3.2.5 代码评审 Agent
- **输入**：代码变更集 + 方案 + 测试结果
- **输出**：`devflow.code_review.v1` JSON + 评审报告
- **功能**：
  - 只读审查（正确性、安全性、规范性）
  - 阻塞问题自动修复循环（最多 1 次重试）
  - 区分阻塞问题和非阻塞建议

#### 3.2.6 交付 Agent
- **输入**：评审通过的变更集
- **输出**：`devflow.delivery.v1` 交付包
- **功能**：
  - 汇总变更、测试证据、评审结论
  - 捕获 Git 状态（分支、HEAD、状态）
  - 生成合并建议，不自动提交/推送/创建 PR

## 4. Human-in-the-Loop 机制

### 4.1 双重审批通道
1. **飞书审批（主通道）**
   - 自动创建第三方审批实例
   - 用户在飞书审批 App 中操作
   - 支持 Approve/Reject 操作
   - 驳回需提供理由

2. **Bot 消息（回退通道）**
   - 通过飞书聊天界面交互
   - `Approve <run_id>` 通过
   - `Reject <run_id>` 驳回
   - 支持前缀匹配，自动处理歧义

### 4.2 质量门控
- **方案设计质量检查**：`quality.ready_for_code_generation` 标记
- **阻塞性审批**：质量不达标时进入 `waiting_approval_with_warnings` 状态
- **强制通过**：`Approve <run_id> --force` 或 `强制通过` 命令

### 4.3 会话管理
- **Session Manager**：基于 `chat_id:sender_id` 的会话状态跟踪
- **消息路由**：上下文关联，支持澄清、追加、恢复等交互模式
- **超时清理**：默认 1800 秒会话超时

## 5. 数据模型设计

### 5.1 版本化 JSON 契约体系

| 契约 | Schema Version | 产出内容 |
|------|---------------|---------|
| 需求分析 | `devflow.requirement.v1` | 结构化需求（含验收标准、质量信号） |
| 方案设计 | `devflow.solution_design.v1` | 技术方案（含变更清单、API 设计） |
| 代码生成 | `devflow.code_generation.v1` | 代码变更集 + diff |
| 测试生成 | `devflow.test_generation.v1` | 测试代码 + 执行结果 + diff |
| 代码评审 | `devflow.code_review.v1` | 评审报告（含问题列表和修复建议） |
| 交付打包 | `devflow.delivery.v1` | 可交付代码变更 + 变更摘要 |
| 检查点 | `devflow.checkpoint.v1` | 审批状态、决策记录 |
| Pipeline 运行 | `devflow.pipeline_run.v1` | 运行状态、阶段产物路径、审计信息 |

## 6. 核心组件设计

### 6.1 管道编排器 (Pipeline Orchestrator)
- **LangGraph 状态图**：定义了 6 个阶段和 2 个检查点的有向工作流
- **检查点恢复**：支持任意点暂停和恢复
- **条件分支**：基于质量门控和审批结果的智能路由
- **错误处理**：优雅降级和重试机制

### 6.2 Agent 架构
每个 Agent 包含：
1. **系统提示**：定义 Agent 角色和能力边界
2. **工具集**：阶段特定的工具访问权限
3. **多轮对话**：支持工具调用的迭代执行
4. **审计日志**：完整的执行轨迹记录
5. **参考文档**：基于阶段的参考文档注入

### 6.3 代码库语义索引
- **Python 支持**：内置 `ast` 模块，零依赖
- **JS/TS 支持**：`tree-sitter` 可选依赖，优雅降级
- **5 种查询**：符号搜索、引用查找、调用链追踪、继承层次、依赖分析
- **增量更新**：SHA256 哈希变更检测
- **存储**：JSON 文件，适用于 ≤ 1000 文件的工作区

### 6.4 多 LLM Provider 支持
- **内置提供商**：火山方舟、百炼、DeepSeek、OpenAI、自定义
- **运行时切换**：CLI/API 级别动态切换
- **统一接口**：OpenAI-compatible Chat Completions
- **令牌使用跟踪**：完整的使用量和成本分析

## 7. 系统特性

### 7.1 飞书原生集成
- **WebSocket 长连接**：通过 lark-cli 1.0.23 建立双向通信
- **事件去重**：防止重复消息触发多次运行
- **PRD 自动创建**：需求分析后自动生成飞书文档
- **交互式卡片**：阶段状态实时推送
- **审批集成**：支持飞书原生审批流

### 7.2 可观测性
- **实时仪表板**：Vanilla HTML/JS/CSS 零构建界面
- **阶段详情**：LLM 推理链、令牌消耗、耗时统计
- **API 文档**：Swagger UI 和 ReDoc 交互式文档
- **审计日志**：完整的工具调用和决策记录

### 7.3 参考文档系统
13 个行业标准文档，按阶段注入 Agent 提示：
- **需求分析阶段**：EARS 语法、NFR 清单
- **方案设计阶段**：ADR 模板、技术选型、分层架构、API 设计、数据库模式、认证流程
- **代码生成阶段**：Git 规范、环境管理、Karpathy 编码原则
- **测试生成阶段**：测试策略、Karpathy 编码原则
- **代码评审阶段**：NFR 清单、发布清单、Karpathy 编码原则

### 7.4 安全与边界控制
- **工作区沙盒**：代码生成 Agent 只能在指定目录内操作
- **受限工具集**：严格控制文件系统访问权限
- **命令白名单**：powershell 工具支持的命令有限集
- **AST 边界检查**：防止越界修改
- **审核日志**：所有操作都被记录和审计

## 8. 部署与运行

### 8.1 系统要求
- Python 3.11+
- 飞书企业自建应用凭证
- LLM API Key（支持任意 OpenAI-compatible 提供商）
- 本地代码库或新建项目空间

### 8.2 快速启动
```powershell
# 1. 安装依赖
pip install uv
npm install -g @larksuite/cli@1.0.23

# 2. 配置凭证
Copy-Item config.example.json config.json
# 编辑 config.json 填入 LLM 和飞书凭证

# 3. 启动服务
uv run devflow serve --host 127.0.0.1 --port 8080

# 4. 启动 Bot 监听
uv run devflow start
```

### 8.3 四种使用模式
1. **飞书 Bot**：在飞书群中直接发送需求消息
2. **REST API**：通过 HTTP 接口触发 Pipeline
3. **CLI 逐步执行**：手动控制每个阶段
4. **一键全栈启动**：同时启动 API、Bot 和前端仪表板

## 10. 总结

DevFlow Engine 是一个完整、可扩展的 AI 驱动研发流程引擎，其核心创新在于：

1. **人机协同的工作流设计**：6 阶段 Pipeline + 2 个检查点的黄金分割点
2. **飞书原生深度集成**：从需求输入到交付的端到端飞书体验
3. **安全的自动化边界**：工具沙盒、AST 边界检查、有界重试
4. **结构化代码理解**：AST 语义索引 + 参考文档系统
5. **企业级可观测性**：完整的审计追踪和可视化监控

该设计平衡了自动化程度和人工控制权，在提升研发效率的同时确保代码质量和安全性，为 AI Native 研发范式提供了可行的工程实现路径。
