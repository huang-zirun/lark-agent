## 概述

发布是软件交付中风险最高的环节。本参考基于 Google SRE 发布工程实践与 Atlassian Release Management 流程，定义发布前、中、后的三层检查清单，确保发布过程可控、可回滚、可观测。

## 发布前检查

### 代码质量

- [ ] Code Review 已完成，无未解决的 Blocker 评论
- [ ] 无已知的 P0/P1 Bug 未修复
- [ ] 代码符合 Lint 规则，无新增 Warning
- [ ] 依赖版本已锁定，无已知高危 CVE

### 测试

- [ ] 单元测试通过，覆盖率达标
- [ ] 集成测试通过
- [ ] E2E 核心路径测试通过
- [ ] 性能测试通过（P95 延迟、吞吐量达标）
- [ ] 回归测试通过（无意外破坏）

### 数据库

- [ ] Migration 脚本已验证（含 Rollback）
- [ ] 大表变更已评估锁表影响
- [ ] 数据迁移脚本已在 Staging 验证
- [ ] Schema 变更向前兼容（旧代码可运行）

### API

- [ ] 无未声明的 Breaking Change
- [ ] API 文档已更新
- [ ] API 版本号已按需递增
- [ ] 新增 API 已有集成测试覆盖

### 配置

- [ ] 新增配置项已在所有环境配置
- [ ] 密钥已轮换或更新
- [ ] Feature Flag 已配置（灰度功能）
- [ ] 配置变更已在 Staging 验证

### 监控

- [ ] 新增指标已接入监控面板
- [ ] 告警规则已配置（SLO/错误率/延迟）
- [ ] 日志格式符合结构化规范
- [ ] 分布式追踪链路完整

### 文档

- [ ] CHANGELOG 已更新
- [ ] 运维手册已更新（如有新操作流程）
- [ ] 用户文档已更新（如有功能变更）

### 回滚计划

- [ ] 回滚方案已文档化
- [ ] 回滚步骤已在 Staging 验证
- [ ] 数据库回滚脚本已准备
- [ ] 回滚决策条件已明确（错误率阈值、响应时间阈值）

## 发布中检查

### 部署顺序

1. 数据库 Migration 先于应用部署
2. 消费者先于生产者部署（兼容性保障）
3. 无状态服务先于有状态服务

### 健康检查

- [ ] 部署后 Health Check 通过
- [ ] 启动日志无 ERROR
- [ ] 就绪探针（Readiness Probe）通过

### 灰度发布

- [ ] Canary 部署至 5% 流量
- [ ] 观察 5-15 分钟，指标正常
- [ ] 逐步扩大至 25% → 50% → 100%
- [ ] 每阶段观察错误率与延迟

## 发布后检查

### 监控验证

- [ ] 错误率未超过阈值（< 1%）
- [ ] P95 延迟未劣化（≤ 基线 120%）
- [ ] 无异常告警触发
- [ ] 业务指标正常（订单量、活跃用户等）

### 用户验证

- [ ] 核心用户路径手动验证通过
- [ ] 客户反馈渠道正常

### 回滚决策

若出现以下情况，立即执行回滚：
- 错误率 > 5% 持续 5 分钟
- P95 延迟 > 基线 200%
- 数据丢失或数据不一致
- 安全漏洞被触发

## Agent 使用指引

1. **清单自动生成**: 根据变更类型（功能/修复/重构/数据库变更）自动生成定制化检查清单，仅包含相关检查项。
2. **发布风险评估**: 分析变更范围，评估发布风险等级（低/中/高），高风险变更建议增加灰度观察时间。
3. **检查项自动验证**: 对可自动化的检查项（测试通过率、Lint 规则、CVE 扫描、API 兼容性）自动执行并标注结果。
4. **回滚方案生成**: 基于变更内容自动生成回滚方案，包含数据库回滚脚本与应用回滚步骤。
