## 概述

非功能需求（NFR）决定系统的质量属性，却常在需求阶段被遗漏。本清单基于 ISO/IEC 25010 产品质量模型、OWASP ASVS 安全验证标准及 Google SRE 实践，提供九大维度的检查项与典型阈值，确保需求与代码层面的 NFR 覆盖完整。

## 九大维度

### 1. Performance（性能）

**关键问题**: 关键操作的响应时间与吞吐量目标是什么？是否存在慢查询或 N+1 问题？

**典型阈值**: P95 延迟 ≤ 200ms（读操作）、≤ 1s（写操作）；吞吐量按业务峰值 1.5 倍设计。

### 2. Scalability（可扩展性）

**关键问题**: 系统水平扩展策略是什么？有无单点瓶颈？数据分片方案是否明确？

**典型阈值**: 支持无状态水平扩展；单节点故障不影响整体可用性；数据层支持分片（Sharding）。

### 3. Availability（可用性）

**关键问题**: SLA 目标是多少？故障恢复时间（RTO）与数据恢复点（RPO）是多少？

**典型阈值**: 核心 API 可用性 ≥ 99.9%；RTO ≤ 5min；RPO ≤ 1min；多可用区部署。

### 4. Security（安全性）

**关键问题**: 认证与授权机制是否完备？输入验证是否覆盖所有入口？敏感数据是否加密存储与传输？

**典型阈值**: 遵循 OWASP ASVS Level 2；传输层 TLS 1.2+；密码 bcrypt/scrypt 存储；API Rate Limiting。

### 5. Reliability（可靠性）

**关键问题**: 是否有重试、熔断、降级策略？关键操作是否有幂等保障？

**典型阈值**: 重试 ≤ 3 次指数退避；熔断阈值 50% 错误率持续 30s；关键写操作幂等。

### 6. Maintainability（可维护性）

**关键问题**: 代码是否分层？模块间耦合度如何？是否有自动化测试与 CI/CD？

**典型阈值**: 圈复杂度 ≤ 15；单元测试覆盖率 ≥ 80%；CI 流水线全自动化。

### 7. Observability（可观测性）

**关键问题**: 是否有结构化日志、指标（Metrics）与分布式追踪（Tracing）？告警规则是否覆盖 SLO？

**典型阈值**: Three Pillars 齐备（Logs/Metrics/Traces）；P99 延迟告警；错误率 > 1% 触发告警。

### 8. Compatibility（兼容性）

**关键问题**: API 是否向后兼容？数据迁移策略是什么？支持哪些客户端版本？

**典型阈值**: API 遵循 SemVer 语义化版本；数据库迁移向前兼容；客户端兼容 N-2 版本。

### 9. Compliance（合规性）

**关键问题**: 是否涉及个人信息（PII）？数据驻留与跨境传输是否合规？审计日志是否完整？

**典型阈值**: 遵循 GDPR/PIPL 等适用法规；PII 加密存储；操作审计日志保留 ≥ 180 天。

## Agent 使用指引

1. **需求阶段**: 扫描需求文档，逐维度检查 NFR 是否被覆盖，输出缺失项清单并建议补充。
2. **代码审查阶段**: 对照各维度阈值，检查代码实现是否满足 NFR 约束（如：是否缺少重试逻辑、是否硬编码密钥、是否缺少索引）。
3. **风险评估**: 对未达标的 NFR 项标注风险等级（高/中/低），并给出改进建议。
4. **阈值定制**: 根据项目类型（内部工具/面向用户/金融级）调整典型阈值，生成项目专属 NFR 规格说明。
