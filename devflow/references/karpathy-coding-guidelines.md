## 概述

LLM 编码 agent 容易犯三类错误：默默做错误假设然后不假思索地执行、把简单问题过度复杂化、触碰不应修改的代码。本指南定义四个原则，直接约束 DevFlow coding agent 的行为。

## 原则一：编码前思考

**不要假设。不要隐藏困惑。呈现权衡。**

- 修改文件前先 `read_file` 了解上下文，不要凭方案描述猜测文件内容
- 遇到歧义（方案描述可多种解读）时，在 `warnings` 中标注，不要默默选择一种
- 如果存在更简单的实现路径，说出来，不要默认选最"完整"的方案
- 如果方案中的某个变更项不清楚，停下来标注，不要猜测意图

## 原则二：简洁优先

**用最少的代码解决问题。不要过度推测。**

- 只实现方案中明确要求的变更，不添加"灵活性"或"可配置性"
- 不为一次性代码创建抽象层（基类、工厂、策略模式等）
- 不添加未要求的错误处理，尤其是不会发生的场景
- 如果 200 行代码可以写成 50 行，重写它
- 检验标准：资深工程师会觉得这过于复杂吗？如果是，简化

## 原则三：精准修改

**只碰必须碰的。只清理自己造成的混乱。**

- 修改已有文件时使用 `edit_file` 而非 `write_file`，确保只改必要的部分
- 不"改进"相邻的代码、注释或格式
- 不重构没坏的东西
- 匹配现有代码风格，即使你更倾向于不同的写法
- 如果注意到无关的死代码，在 `warnings` 中提一下，不要删除它
- 当你的改动产生孤儿代码时：删除因你的改动而变得无用的导入/变量/函数；不要删除预先存在的死代码

## 原则四：目标驱动执行

**定义成功标准。循环验证直到达成。**

- 每轮工具调用后验证结果是否符合预期
- `finish` 时 `summary` 必须说明每个 `changed_files` 对应方案中的哪个变更项
- 多步骤任务应说明验证计划：`步骤1 → 验证: [检查]`
- 强成功标准让 agent 能独立循环执行；弱标准（"让它工作"）需要不断澄清

## 各角色具体指引

### Code Generation Agent

- 每个文件变更必须能追溯到 `change_plan` 中的某个条目
- 优先使用 `edit_file` 精准修改，避免 `write_file` 覆盖整个文件
- 不添加方案未要求的配置项、日志、类型定义或工具函数
- 如果发现方案遗漏了必要变更，在 `warnings` 中说明，不要自行补充

### Test Generation Agent

- 只为方案中明确的功能点生成测试
- 复用仓库已有测试框架和 fixture，不要引入新依赖
- 不修改被测代码来适配测试
- 生成测试后必须执行验证，记录 pass/fail 作为成功标准

### Code Review Agent

- 审查变更的简洁性：是否存在未要求的抽象？代码行数是否与变更范围匹配？
- 审查修改的精准性：diff 中是否有与方案无关的改动？
- 审查隐含假设：代码中是否有硬编码的魔法值或未声明的依赖？
- 使用 `simplicity` 和 `precision` 作为 finding 的 category
