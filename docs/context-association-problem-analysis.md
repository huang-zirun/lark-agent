# DevFlow Engine 上下文关联问题分析报告

## 1. 问题概述

### 现象描述
用户在飞书机器人对话中连续发送需求相关信息时，系统错误地将后续补充信息识别为**全新独立需求**，而非**当前需求的补充/细化**。

### 具体案例

| 时间 | 用户输入 | 系统识别 | 预期行为 | 实际行为 |
|------|---------|---------|---------|---------|
| 08:21 | "设计一款贪吃蛇游戏" | 新需求 | 启动需求分析 | ✅ 正确 |
| - | "春天主题" | 澄清答案 | 追加到当前需求 | ✅ 正确 |
| - | "美丽花花" | 澄清答案 | 追加到当前需求 | ✅ 正确 |
| - | "绿色草草" | 补充信息 | 追加到当前需求 | ❌ **错误识别为新需求** |

---

## 2. 实验数据

### 实验 1：贪吃蛇游戏需求

```yaml
运行 ID: 20260506T002115Z-om_x100b509cd9f9fca4b2767e2b6d7ba95-be1366ef
用户输入: "设计一款贪吃蛇游戏"
source_id: om_x100b509cd9f9fca4b2767e2b6d7ba95
chat_id: oc_dd5a46c3b36066e94cde1a330c062fd6
sender_id: ou_c29c359577e0cbb6377b361c5d883630
状态: blocked（缺少仓库上下文）
```

**关键发现**：`open_questions[0].answer` 字段已记录 `"春天主题\n美丽花花"`

### 实验 2：绿色草草（错误识别）

```yaml
运行 ID: 20260506T002329Z-om_x100b509cd6e590a0b4cfc465b34abc6-62f3021e
用户输入: "绿色草草"
source_id: om_x100b509cd6e590a0b4cfc465b34abc6  # ← 完全不同的 ID
chat_id: oc_dd5a46c3b36066e94cde1a330c062fd6   # ← 相同的对话
sender_id: ou_c29c359577e0cbb6377b361c5d883630 # ← 相同的发送者
状态: waiting_clarification（等待澄清）
```

**关键发现**：系统生成了全新的 `source_id`，完全独立于实验 1

## 3. 根本原因分析

### 3.1 架构层面：消息隔离机制过于严格

```
┌─────────────────────────────────────────────────────────────┐
│                    当前架构流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   飞书消息 A ──→ 生成 source_id_A ──→ 创建 Run A            │
│        │                                                    │
│        │   （消息间无关联）                                  │
│        ▼                                                    │
│   飞书消息 B ──→ 生成 source_id_B ──→ 创建 Run B            │
│        │                                                    │
│        │   （即使同一对话、同一用户、短时间内）               │
│        ▼                                                    │
│   飞书消息 C ──→ 生成 source_id_C ──→ 创建 Run C            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**问题**：每条飞书消息都被视为独立事件，系统没有建立消息间的逻辑关联。

### 3.2 逻辑层面：缺乏会话状态管理

| 维度 | 实验 1 | 实验 2 | 是否相同 |
|------|--------|--------|---------|
| chat_id | `oc_dd5a46c3b36066e94cde1a330c062fd6` | `oc_dd5a46c3b36066e94cde1a330c062fd6` | ✅ 是 |
| sender_id | `ou_c29c359577e0cbb6377b361c5d883630` | `ou_c29c359577e0cbb6377b361c5d883630` | ✅ 是 |
| 时间差 | - | ~2 分钟 | ✅ 短间隔 |
| source_id | `om_x100b509cd9f9fca4b2767e2b6d7ba95` | `om_x100b509cd6e590a0b4cfc465b34abc6` | ❌ 不同 |

**问题**：系统没有利用 `chat_id` + `sender_id` + `时间窗口` 来维护"活跃会话"状态。

### 3.3 语义层面：缺乏补充信息识别能力

```json
// 实验 2 生成的需求分析结果
{
  "normalized_requirement": {
    "title": "绿色草草（需求信息严重缺失）",
    "background": ["当前无有效背景信息，无法提取相关内容"],
    "target_users": ["当前无有效信息，无法识别目标用户"],
    "problem": ["当前无有效信息，无法提取业务痛点"]
  },
  "open_questions": [
    {
      "field": "scope",
      "question": "'绿色草草'具体指代的内容是什么？是产品名称、功能模块、业务活动还是其他业务对象？"
    }
  ]
}
```

**问题**：系统无法识别 "绿色草草" 与 "春天主题"、"美丽花花" 的语义关联（都是视觉风格描述）。

## 4. 问题影响

### 4.1 用户体验层面

```
用户视角的对话流程：
─────────────────────────────────────────
用户: 设计一款贪吃蛇游戏
AI:  [需求分析] 进行中... (0/6)

用户: 春天主题
AI:  已追加到当前需求中 ✓

用户: 美丽花花
AI:  已追加到当前需求中 ✓

用户: 绿色草草
AI:  [需求分析] 进行中... (0/6)  ← 用户困惑：怎么又从头开始？
     需求待澄清：
     Q1: "绿色草草"具体指代的内容是什么？
     ...
─────────────────────────────────────────
```

**用户感受**：
- 对话连贯性被打破
- 重复回答相似问题
- 对 AI 的理解能力产生质疑

### 4.2 系统资源层面

| 指标 | 实验 1 | 实验 2 | 浪费情况 |
|------|--------|--------|---------|
| Token 消耗 | 6,995 | 5,436 | 实验 2 完全无意义 |
| 耗时 | 59,837ms | 34,658ms | ~35 秒用户等待 |
| 存储 | 完整需求产物 | "信息严重缺失"产物 | 无效数据存储 |
