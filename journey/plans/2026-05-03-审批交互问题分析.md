# DevFlow 审批交互问题分析：用户无法通过按钮/审批/拒绝回复

## 问题现象

当需求经过 `requirement_intake` 和 `solution_design` 两个节点后，系统向用户发送了技术方案评审卡片（见截图）。卡片中包含：
- 方案摘要、运行 ID、风险等级
- 方案 JSON 和文档路径
- 文件变更预览
- **同意：`Approve {run_id}`**
- **拒绝：`Reject {run_id}`**

**用户反馈：收到卡片后无法回复，不能通过按钮、审批或拒绝并附带理由。**

---

## 根本原因分析

### 1. 当前实现使用的是"纯文本命令"交互模式，而非真正的交互式卡片按钮

查看 [checkpoint.py](file:///d:/lark/devflow/checkpoint.py#L147-L149) 中的卡片构建逻辑：

```python
body = "\n".join(
    [
        f"**运行 ID**：`{run_id}`",
        ...
        f"同意：`Approve {run_id}`",
        f"拒绝：`Reject {run_id}`",
    ]
)
```

卡片中的"同意/拒绝"只是**纯文本提示**，告诉用户需要手动输入 `Approve 20260503T123303Z-...` 或 `Reject 20260503T123303Z-...` 这样的文本来回复。

**这不是可点击的按钮。**

### 2. 飞书消息卡片中不支持真正的交互按钮（当前 lark-cli 1.0.23 限制）

查看 [design.md](file:///d:/lark/journey/design.md#L41) 中的关键决策：

> The first checkpoint uses bot message replies as the v1 action channel because `lark-cli` 1.0.23 exposes IM message events but not card button callback events. `Approve <run_id>` records approval and a continuation request; `Reject <run_id>` starts a two-turn rejection flow that captures the next reply as the reject reason and reruns `solution_design`.

**核心限制**：`lark-cli` 1.0.23 只暴露了 IM 消息接收事件 (`im.message.receive_v1`)，**没有暴露卡片按钮回调事件**。因此系统无法监听用户点击卡片按钮的动作。

### 3. 用户期望 vs 实际实现的差距

| 用户期望 | 实际实现 | 差距 |
|---------|---------|------|
| 点击"同意"按钮直接审批 | 需要手动输入 `Approve {run_id}` | 没有可点击按钮 |
| 点击"拒绝"按钮并输入理由 | 需要先输入 `Reject {run_id}`，再输入理由 | 两回合交互，不直观 |
| 在审批界面操作 | 在聊天窗口输入文本命令 | 交互渠道不一致 |

---

## 选定方案：方案 A — 飞书审批流程

经过调研，确定使用**飞书审批**替代当前的 IM 文本命令交互方式。

### 为什么选方案 A

1. **符合用户直觉**：用户在飞书"审批"应用中操作，看到待办事项，点击同意/拒绝
2. **拒绝时可附带理由**：飞书审批原生支持审批意见（comment），无需两回合交互
3. **审批记录天然留痕**：所有操作记录在飞书审批系统中，便于追溯
4. **lark-cli 已暴露相关 API**：可以查询任务、同意、拒绝

### 飞书审批 API 能力调研结果

| API | 功能 | 风险等级 | 所需 Scope |
|-----|------|---------|-----------|
| `approval.instances.get` | 获取审批实例详情 | read | `approval:instance:read` |
| `approval.tasks.query` | 查询用户任务列表 | read | `approval:task:read` |
| `approval.tasks.approve` | 同意审批任务 | **high-risk-write** | `approval:task:write` |
| `approval.tasks.reject` | 拒绝审批任务 | **high-risk-write** | `approval:task:write` |

**重要发现**：
- `lark-cli` **没有** `approval.instances.create` 命令，无法通过 CLI 创建审批实例
- 需要使用**原生 OpenAPI** 创建审批实例（`POST /open-apis/approval/v4/instances`）
- 或者使用**第三方审批**模式（External Approval），通过 `lark-openapi-explorer` 查找具体接口

## 实施方案 A 的详细计划

### 阶段 1：审批模板配置（一次性）

1. 在飞书管理后台创建"DevFlow 技术方案评审"审批模板
2. 模板包含：
   - 标题（自动填充：技术方案评审 - {run_id}）
   - 方案摘要（多行文本）
   - 方案文档链接（链接类型）
   - 风险等级（单选：low/medium/high）
3. 记录审批模板的 `definition_code`
4. 在 `config.json` 中配置 `approval.definition_code`

### 阶段 2：代码改造

#### 2.1 新增审批模块 `devflow/approval.py`

功能：
- `create_approval_instance(run_payload, solution_artifact, user_open_id)` → 创建审批实例，返回 `instance_code`
- `query_approval_status(instance_code)` → 查询审批状态
- `parse_approval_result(instance_detail)` → 解析审批结果（同意/拒绝/理由）

#### 2.2 修改 `devflow/checkpoint.py`

- 保留 `build_solution_review_card`（作为降级方案）
- 新增 `build_solution_review_approval_request(run_payload, solution_artifact)` → 构建审批实例创建参数

#### 2.3 修改 `devflow/pipeline.py`

在 `publish_solution_review_checkpoint` 中：
1. 尝试创建飞书审批实例
2. 如果成功，发送 IM 消息告知用户"已发起审批，请在审批应用中处理"
3. 如果失败，降级为现有卡片方案

#### 2.4 新增审批状态轮询/事件监听

**选项 A：轮询模式**
- `devflow checkpoint poll` 命令：轮询所有 `waiting_approval` 状态的 checkpoint
- 定时查询 `approval.instances.get` 获取状态
- 状态变化时自动更新 checkpoint 并触发后续流程

**选项 B：事件订阅模式**
- 订阅飞书审批事件（如果 lark-cli 支持）
- 或者使用 `lark-openapi-explorer` 查找审批事件订阅方式

### 阶段 3：交互流程改造

**新流程**：

```
用户发送需求
  → requirement_intake（需求分析）
  → solution_design（技术方案生成）
  → 创建飞书审批实例（DevFlow 技术方案评审）
  → 用户收到审批待办通知
  → 用户在飞书审批应用中查看并操作（同意/拒绝+理由）
  → 系统轮询/接收事件获取结果
  → 同意 → 进入 code_generation
  → 拒绝 → 根据理由重做 solution_design
```

**降级流程**（审批创建失败时）：
```
  → 发送现有交互式卡片
  → 用户回复 Approve/Reject 命令
  → 现有逻辑处理
```

### 阶段 4：权限与配置

1. 在飞书开发者后台为应用开通以下 scope：
   - `approval:instance:read`
   - `approval:task:read`
   - `approval:task:write`
   - `approval:instance:write`（如需创建实例）
2. 更新 `config.example.json`，添加 `approval` 配置段：
   ```json
   {
     "approval": {
       "definition_code": "7C468A54-8745-2245-9675-08B7C63E7A85",
       "enabled": true,
       "poll_interval_seconds": 60
     }
   }
   ```

## 风险与限制

1. **lark-cli 不支持创建审批实例**：需要使用原生 OpenAPI 或第三方审批模式
2. **需要预配置审批模板**：用户需要在飞书管理后台手动创建模板并获取 `definition_code`
3. **轮询模式有延迟**：审批状态不是实时同步的，取决于轮询间隔
4. **用户需要离开聊天上下文**：审批操作在"审批"应用中完成，不在当前聊天窗口
5. **high-risk-write 操作**：`tasks.approve` 和 `tasks.reject` 是高风险操作，需要 `--yes` 确认（如果通过 CLI 调用）

---

## 替代建议：如果方案 A 实施成本过高

如果飞书审批的模板配置和实例创建过于复杂，可以考虑：

1. **使用 lark-task（飞书任务）**：创建任务代替审批，用户完成任务/评论来交互
2. **优化现有卡片文案**：明确提示用户"请回复以下文字"，降低误解
3. **调研 lark-cli 新版本**：查看是否有卡片按钮回调事件支持
