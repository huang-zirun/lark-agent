# 计划：借鉴 PRD Skill 提示词设计提升需求分析 Agent 效果

## 背景

`d:\lark\docs\SKILL.md` 是一个为 Claude Code 设计的产品经理 Skill，提供了三模式（生成/澄清/扩展）的 PRD 工作流。现有 DevFlow Engine 的 `ProductRequirementAnalyst`（位于 `devflow/intake/`）是流水线第一个节点，角色定位为"资深产品经理"，但当前实现存在明显的改进空间。

**核心约束**：需求分析 Agent 在流水线中只接受原始输入，是单次分析节点，不引入 Clarify 模式或多轮交互。改进聚焦于**在单轮分析中借鉴 SKILL.md 的设计精髓**来提升产物质量。

---

## 一、SKILL.md 设计亮点分析（与单轮分析相关的部分）

### 1.1 结构化扫描框架 — 系统化识别缺失与模糊

Clarify 模式定义了 **10 大分类**的结构化扫描框架：

1. 功能范围与行为
2. 领域与数据模型
3. 交互与 UX 流程
4. 非功能属性
5. 集成与外部依赖
6. 边缘情况
7. 约束与权衡
8. 术语一致性
9. 完成信号
10. 模糊占位符

**可借鉴点**：将扫描框架融入 Generate 阶段，让 LLM 在生成产物时**同时**按分类评估信息完整度，而非事后补充 `open_questions`。

### 1.2 输入溯源机制 — 需求可追溯

`## 输入历史 (Input History)` 章节逐字保留用户原话，不改写、不翻译、不压缩。

**可借鉴点**：在产物中保留用户原始输入的完整记录，确保下游 Agent 和人工审阅时能回溯"用户到底说了什么"。

### 1.3 质量自检闭环 — 不落盘但必执行

Generate 模式写完后对照"质量清单"逐项检查，最多迭代 3 次。质量清单分三个维度：

- **内容质量**：不含实现细节、聚焦用户价值、非技术干系人可读
- **需求完整性**：`[NEEDS CLARIFICATION]` ≤ 3、需求可测试、成功标准可衡量
- **功能就绪度**：验收标准清晰、用户场景覆盖主流程

**可借鉴点**：在 LLM Prompt 中加入质量自检指令，让模型在输出前自行检查产物质量。

### 1.4 用户故事模板 — 优先级 + 独立测试 + Given/When/Then

SKILL.md 的用户故事包含：
- 优先级（P1/P2/P3）及原因
- 独立测试描述
- Given/When/Then 验收场景
- 边界情况

**可借鉴点**：将 `user_scenarios` 从字符串数组升级为结构化对象，增加优先级和验收场景。

### 1.5 模糊占位符识别 — 识别"有内容但含糊"

SKILL.md 明确要求识别 "robust"/"intuitive" 等无量化的形容词，以及 TODO 等占位符。

**可借鉴点**：在分析过程中检测模糊表述，在 `open_questions` 中生成量化追问，而非只检查字段是否为空。

### 1.6 合理推测 + 限制澄清标记

SKILL.md 的 Generate 流程要求：
- 对没明说的细节，基于行业标准填默认值，在"假设"章节记录
- 最多保留 3 个 `[NEEDS CLARIFICATION]` 标记，按优先级排序
- 有合理默认值的一律不标

**可借鉴点**：让 Agent 主动推断而非被动留空，减少不必要的 `open_questions`，同时确保关键模糊点被标记。

## 二、现有 ProductRequirementAnalyst 的差距

### 2.1 缺乏结构化扫描框架

**现状**：启发式模式用关键词匹配提取 12 个字段，LLM 模式让模型自由输出 5 个顶层字段。

**差距**：
- 没有 10 大分类扫描框架，无法系统化地识别"哪类信息缺失"
- `open_questions` 的生成逻辑过于简单（只检查 5 个字段是否为空），无法识别"有内容但模糊"的情况
- 缺少对非功能属性、边缘情况、术语一致性等维度的覆盖

### 2.2 质量评估过于粗糙

**现状**：`build_quality()` 只计算 4 个必填字段 + 验收标准的覆盖率，以及 `open_questions` 数量占比。

**差距**：
- 没有区分"缺失"和"模糊"——字段有内容但表述含糊（如"系统要快"）不会被标记
- 没有 SKILL.md 的质量清单（内容质量/需求完整性/功能就绪度三维检查）
- `ready_for_next_stage` 的阈值硬编码（0.6/0.6），缺乏语义层面的判断

### 2.3 无输入溯源机制

**现状**：产物中只有 `source.reference` 和 `source.safe_summary`，没有逐字保留用户原始输入。

**差距**：
- 需求经过 LLM 改写后，原始表述可能丢失关键细节
- 下游 Agent 无法回溯"用户到底说了什么"

### 2.4 用户故事缺乏优先级和独立测试性

**现状**：`user_scenarios` 只是字符串数组，没有优先级（P1/P2/P3）、没有独立测试描述、没有验收场景（Given/When/Then）。

**差距**：
- 无法区分核心场景和边缘场景的优先级
- 下游 Agent 无法判断哪些场景必须先实现

### 2.5 Prompt 过于简短，缺少工作流指引

**现状**：`PRODUCT_REQUIREMENT_ANALYST_PROMPT` 只有 28 行，列出了 6 项核心能力和输出契约。

**差距**：
- 没有具体的分析步骤指引
- 没有质量自检指令
- 没有对模糊占位符的识别指引
- 没有 EARS 语法的具体示例
- 没有合理推测 + 限制澄清标记的规则

### 2.6 缺乏功能需求编号和成功标准

**现状**：产物中没有功能需求（FR-NNN）和成功标准（SC-NNN）的编号体系。

**差距**：
- SKILL.md 的 Generate 模式要求每条功能需求可测试、无歧义，每条成功标准可衡量、技术无关
- 现有产物只有 `acceptance_criteria`（验收标准），缺少功能需求分解和成功标准定义

## 三、改进方案

### 3.1 增强 System Prompt（高优先级，不改 schema）

**目标**：将 SKILL.md 的工作流指引融入 Agent 的 System Prompt，提升单轮分析的质量。

**具体改动** — 修改 `devflow/intake/prompt.py`：

在 `PRODUCT_REQUIREMENT_ANALYST_PROMPT` 中增加以下内容：

1. **明确的分析步骤**：
   ```
   分析步骤：
   1. 解析输入：从需求描述中提取参与者(actors)、操作(actions)、数据(data)、约束(constraints)
   2. 合理推测：对没明说的细节，基于行业标准和常见模式填默认值，在 assumptions 中记录
   3. 限制澄清标记：最多保留 3 个 [NEEDS CLARIFICATION] 标记，按优先级：范围 > 安全/隐私 > 用户体验 > 技术细节。有合理默认值的一律不标
   4. 用户故事排序：按重要性排序(P1/P2/P3)，每个故事必须可独立测试、独立交付价值
   5. 验收标准：使用 EARS 语法模式，确保需求无歧义
   6. 质量自检：输出前对照质量清单逐项检查
   ```

2. **结构化扫描框架**（融入分析过程，不是独立模式）：
   ```
   在分析需求时，按以下维度系统评估信息完整度：
   - 功能范围与行为：核心目标、成功标准、超出范围声明、用户角色区分
   - 领域与数据模型：实体/属性/关系、唯一性、生命周期
   - 交互与 UX 流程：关键旅程、错误/空/加载态
   - 非功能属性：性能、安全隐私、合规
   - 集成与外部依赖：外部服务、数据导入导出
   - 边缘情况：负面场景、限流、冲突解决
   - 模糊占位符：识别"robust"/"intuitive"/"快速"/"稳定"/"高效"等无量化的形容词，在 open_questions 中生成量化追问
   ```

3. **模糊占位符识别指引**：
   ```
   模糊表述检测：当需求中出现以下模式时，必须在 open_questions 中标记并建议量化方向：
   - 无量化的性能描述："快速"/"高效"/"稳定"/"robust"/"fast"/"reliable"
   - 无边界的范围描述："等"/"等等"/"etc."/"and so on"
   - 主观评价："友好"/"直观"/"intuitive"/"user-friendly"
   - 未定义的代词："它"/"该系统"/"the system" 指代不清时
   ```

4. **EARS 语法示例**：
   ```
   EARS 语法模式示例（用于编写验收标准）：
   - 无条件型："系统必须 [具体能力]"
   - 事件驱动型："当 [事件发生] 时，系统必须 [响应]"
   - 不需要型："系统不得 [禁止行为]"
   - 状态驱动型："当 [系统处于某状态] 时，系统必须 [行为]"
   - 可选型："当 [可选功能被启用] 时，系统必须 [行为]"
   ```

5. **质量自检指令**（不落盘但必须执行）：
   ```
   质量自检（输出前必须逐项检查，检查结果不写入产物）：
   - 内容质量：不含实现细节(语言/框架/API)、聚焦用户价值、非技术干系人可读
   - 需求完整性：[NEEDS CLARIFICATION] ≤ 3、每条验收标准可测试无歧义、成功标准可衡量技术无关
   - 功能就绪度：验收标准覆盖主流程、用户场景有优先级区分
   如不满足，调整输出后再返回。
   ```

### 3.2 增强用户故事结构（中优先级，需 schema 变更）

**目标**：用户故事包含优先级、独立测试性和 Given/When/Then 验收场景。

**具体改动**：

1. **在 `devflow.requirement.v1` schema 中将 `user_scenarios` 升级为 `user_stories`**：
   ```json
   {
     "user_stories": [
       {
         "id": "US-001",
         "title": "简要标题",
         "priority": "P1",
         "description": "作为某类角色，我希望完成某个动作，以便获得某个收益",
         "priority_reason": "为什么是 P1",
         "independent_test": "如何独立验证这个故事的价值",
         "acceptance_scenarios": [
           {
             "given": "初始状态",
             "when": "用户操作",
             "then": "预期结果"
           }
         ]
       }
     ]
   }
   ```

2. **保持向后兼容**：`user_scenarios` 保留为可选字段，从 `user_stories` 的 `description` 自动提取生成

3. **在 LLM User Prompt 中明确要求按此结构输出用户故事**

4. **更新 `prd.py` 渲染逻辑以支持新的用户故事格式**

5. **更新 `analyzer.py` 中的 `normalize_llm_analysis()` 以解析新结构**

### 3.3 引入输入溯源机制（中优先级，需 schema 变更）

**目标**：保留用户原始输入，确保需求可追溯。

**具体改动**：

1. **在 `devflow.requirement.v1` schema 中新增 `input_history` 字段**：
   ```json
   {
     "input_history": [
       {
         "timestamp": "2026-05-06T14:32:00Z",
         "mode": "generate",
         "trigger": "首次创建需求",
         "raw_input": "用户原始输入逐字保留..."
       }
     ]
   }
   ```

2. **在 `build_artifact_payload()` 中初始化 `input_history`**：
   - 从 `source.content` 逐字保留用户原始输入
   - 记录时间戳、模式、触发原因

3. **在 `prd.py` 的 `render_prd_markdown()` 中渲染输入历史章节**

### 3.4 增强质量评估（高优先级，部分不需 schema 变更）

**目标**：从简单的覆盖率计算升级为三维质量评估。

**具体改动**：

1. **在 LLM Prompt 中增加质量评估维度指引**（不改 schema）：
   - 让 LLM 在 `quality` 字段中输出更丰富的评估信息
   - 新增 `quality.dimensions` 子字段：
     ```json
     {
       "quality": {
         "completeness_score": 0.8,
         "ambiguity_score": 0.3,
         "ready_for_next_stage": true,
         "warnings": [],
         "dimensions": {
           "content_quality": "clear",
           "requirement_completeness": "partial",
           "functional_readiness": "clear"
         }
       }
     }
     ```

2. **增强启发式模式的 `build_quality()` 函数**：
   - 新增模糊表述检测（关键词匹配 + 正则）
   - 区分"缺失"和"模糊"两种状态
   - `dimensions` 字段为可选，不影响下游兼容性

3. **增强 `build_open_questions()` 的生成逻辑**：
   - 当前：只检查 5 个字段是否为空
   - 改进：结合模糊表述检测，识别"有内容但含糊"的情况
   - 每个问题附带 `suggested_answer`（建议答案）和 `reasoning`（推理依据）

### 3.5 新增功能需求和成功标准（低优先级，后续迭代）

**目标**：在产物中增加 FR-NNN 功能需求和 SC-NNN 成功标准的编号体系。

**具体改动**：

1. **在 `devflow.requirement.v1` schema 中新增 `functional_requirements` 和 `success_criteria` 字段**
2. **FR-NNN 必须可测试、无歧义**
3. **SC-NNN 必须可衡量、技术无关、以用户/业务视角描述**
4. **更新 `prd.py` 渲染逻辑**

## 四、实施优先级与依赖关系

```
3.1 增强 System Prompt ──────────────┐
                                     ├─→ 3.4 增强质量评估（LLM 部分）
3.4 增强质量评估（启发式部分） ──────┘
         │
         ▼
3.2 增强用户故事结构 ───────────────┤
                                     ├─→ 3.3 输入溯源机制
3.3 输入溯源机制 ───────────────────┘
         │
         ▼
3.5 功能需求和成功标准（后续迭代）
```

**建议实施顺序**：
1. **Phase 1**（核心提升，不改 schema）：3.1 → 3.4（LLM 部分） — 增强 Prompt 和 LLM 输出质量
2. **Phase 2**（结构升级，需 schema 变更）：3.2 → 3.3 — 升级用户故事结构和输入溯源
3. **Phase 3**（启发式增强）：3.4（启发式部分） — 增强启发式模式的质量评估和模糊检测
4. **Phase 4**（后续迭代）：3.5 — 功能需求和成功标准编号体系

## 五、关键风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Schema 变更导致下游 Agent 不兼容 | 高 | 采用增量字段（新增字段有默认值），不删除现有字段；`user_scenarios` 保留为兼容字段 |
| Prompt 增长导致 Token 消耗增加 | 中 | System Prompt 增加约 800 token，在可接受范围内；质量自检在单轮内完成，不额外调用 LLM |
| LLM 输出结构变化导致解析失败 | 中 | `normalize_llm_analysis()` 已有健壮的容错逻辑，新增字段用 `.get()` 带默认值 |
| 用户故事结构升级影响 PRD 渲染 | 低 | `prd.py` 同时支持旧格式（字符串数组）和新格式（对象数组） |

## 六、预期效果

1. **需求产物质量提升**：结构化扫描框架 + 质量自检 + 模糊表述检测，减少"有内容但含糊"的问题
2. **需求可追溯性增强**：输入溯源机制确保每个需求决策都能回溯到用户原话
3. **用户故事可交付性提升**：优先级 + 独立测试 + Given/When/Then，使下游 Agent 能更好地理解和实现需求
4. **open_questions 质量提升**：从"字段为空才提问"升级为"识别模糊表述并生成量化追问"
