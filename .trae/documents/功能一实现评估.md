# 功能一实现情况全面评估

基于 `docs/功能一.md` 的需求定义与当前代码库实现状态的对比分析。

> 评估范围：排除最后一个交付节点（delivery/交付集成），聚焦前五个阶段及跨阶段需求。

---

## 一、Must-have 需求逐项评估

### 1. Pipeline 引擎

| 子需求 | 状态 | 说明 |
|--------|------|------|
| 阶段的定义、排序与依赖管理 | ⚠️ 部分实现 | `pipeline.py` 中 `STAGE_NAMES` 硬编码了6个阶段顺序，但**无显式依赖管理机制**——阶段间依赖隐含在顺序执行中，不支持 DAG 依赖图或并行阶段。阶段定义不可配置（无法增删改阶段）。 |
| 每个阶段绑定一个或多个 AI Agent 执行 | ✅ 已实现 | 5个阶段各有对应 Agent：`intake.analyzer`、`solution.designer`、`code.agent`、`test.agent`、`review.agent`。但**同一阶段不支持多 Agent**（需求要求"一个或多个"）。 |
| 阶段间的数据流转 | ✅ 已实现 | 上一阶段的 artifact 路径写入 `run.json`，下一阶段通过 `load_*_artifact` 读取。数据流转链路完整：requirement → solution → code-generation → test-generation → code-review。 |
| Pipeline 的启动、暂停、恢复、终止等生命周期管理 | ⚠️ 部分实现 | 启动 ✅（`devflow start`）、恢复 ✅（`checkpoint resume`）、暂停 ❌（无显式 pause 命令/API）、终止 ❌（无 cancel/terminate 命令/API）。当前只有 `blocked` 状态隐式暂停，无主动暂停能力。 |

**未按期实现的关键项：**
- ❌ **Pipeline 可配置性**：阶段定义硬编码，不支持用户自定义阶段组合、排序或增删。需求明确要求"可配置的流水线引擎"。
- ❌ **阶段依赖管理**：无 DAG 依赖图，不支持并行阶段执行。
- ❌ **Pipeline 生命周期不完整**：缺少 pause 和 terminate 操作。

### 2. Agent 编排与执行

| 子需求 | 状态 | 说明 |
|--------|------|------|
| 每个 Agent 具备明确的角色定义（System Prompt） | ✅ 已实现 | 各 Agent 模块均有独立 `prompt.py`（`intake/prompt.py`、`solution/prompt.py`、`code/prompt.py`、`test/prompt.py`、`review/prompt.py`），定义了 System Prompt。 |
| 输入输出契约 | ✅ 已实现 | 各阶段有稳定的 JSON schema version（`devflow.requirement.v1`、`devflow.solution_design.v1`、`devflow.code_generation.v1`、`devflow.test_generation.v1`、`devflow.code_review.v1`）。 |
| Agent 能感知代码库上下文 | ✅ 已实现 | 通过 `workspace` 配置提供目录路径，`solution.designer` 读取代码库上下文，`code.agent` 和 `test.agent` 通过 `read_file`/`glob_search`/`grep_search` 工具感知代码库。 |
| LLM Provider 可配置（至少 2 个） | ✅ 已实现 | 支持 5 个 provider：`ark`（火山引擎）、`bailian`（阿里百炼）、`deepseek`、`openai`、`custom`。 |
| 运行时可切换 | ⚠️ 部分实现 | Provider 通过 `config.json` 静态配置，**不支持单次 Pipeline 运行中动态切换** provider。需修改配置文件后重启才能切换。 |

**未按期实现的关键项：**
- ❌ **运行时动态切换 LLM Provider**：当前仅支持配置文件级别的静态切换，不满足"运行时可切换"需求。

### 3. Human-in-the-Loop 检查点

| 子需求 | 状态 | 说明 |
|--------|------|------|
| 至少 2 个人工检查点 | ✅ 已实现 | 检查点1：方案设计审批（`solution_design` 阶段后）；检查点2：代码评审确认（`code_review` 阶段后）。 |
| Approve（继续）/ Reject（回退到上一阶段重做，携带理由） | ✅ 已实现 | `approve_checkpoint_run` 和 `reject_checkpoint_run` 完整实现。Reject 方案设计会重跑 `solution_design`；Reject 代码评审会触发一次自动修复。 |
| 清晰的检查点 UI 或 API | ⚠️ 部分实现 | 有 Lark 交互卡片 UI（`build_solution_review_card`、`build_code_review_card`）和 CLI 命令（`checkpoint decide`、`checkpoint poll`），但**无独立 Web UI**，依赖飞书客户端。 |

**未按期实现的关键项：**
- ⚠️ 检查点交互渠道依赖飞书 IM/审批，无独立 Web UI 或 REST API 供非飞书用户使用。

### 4. API-First 架构

| 子需求 | 状态 | 说明 |
|--------|------|------|
| 所有核心操作通过 RESTful API 暴露 | ❌ 未实现 | 当前仅有 CLI 接口（`devflow start`、`devflow checkpoint decide` 等），**无 HTTP/REST API 服务器**。 |
| Pipeline CRUD | ❌ 未实现 | 无创建/读取/更新/删除 Pipeline 的 API。Pipeline 运行记录仅以文件形式存储在 `artifacts/runs/`。 |
| 执行触发 | ❌ 未实现 | 无 REST API 触发 Pipeline 执行，仅能通过 CLI 或飞书 bot 事件触发。 |
| 状态查询 | ❌ 未实现 | 无 REST API 查询 Pipeline 运行状态，仅能通过读取本地 JSON 文件查看。 |
| 检查点操作 | ❌ 未实现 | 无 REST API 执行检查点操作（approve/reject），仅能通过 CLI 或飞书消息。 |
| API 文档（Swagger/OpenAPI） | ❌ 未实现 | 无任何 API 文档。 |

**未按期实现的关键项：**
- ❌ **整个 RESTful API 层未实现**——这是功能一的核心架构要求之一，当前完全缺失。
- ❌ **无 API 文档**。

## 五、结论

当前项目在 **Agent 执行层面** 已基本完成功能一前五个阶段的实现（需求分析 → 方案设计 → 代码生成 → 测试生成 → 代码评审），99 个测试通过，核心 Pipeline 数据流转链路完整。

但在 **平台架构层面** 存在严重缺失：

1. **RESTful API 层完全缺失**（Must-have #4）— 这是功能一"API-First 架构"的核心要求，当前仅提供 CLI 接口。
2. **Pipeline 可配置性不足**（Must-have #1）— 阶段定义硬编码，不支持用户自定义流水线。
3. **生命周期管理不完整**（Must-have #1）— 缺少 pause/terminate 操作。

这三个缺失项构成了功能一 Must-have 需求的主要差距，需要在后续迭代中优先补齐。
