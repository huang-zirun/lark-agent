## Must-have 逐项复查

### 1. Pipeline 引擎

| 子需求 | 旧状态 | 新状态 | 说明 |
|---|---|---|---|
| 阶段的定义、排序与依赖管理 | ⚠️ 部分实现 | ✅ 已实现 | `pipeline_config.py` 提供完整的阶段配置解析、依赖声明、DFS 循环检测。支持通过 API `stages` 参数自定义阶段列表（重排/删减） |
| 每个阶段绑定一个或多个 AI Agent 执行 | ✅ 已实现 | ⚠️ 部分实现 | 6 个阶段各有对应 Agent，但**同阶段仍不支持多 Agent**。`BUILT_IN_STAGE_AGENTS` 白名单限制了 Agent 绑定，只能使用内置 Agent |
| 阶段间的数据流转 | ✅ 已实现 | ✅ 已实现 | 无变化，数据流转链路完整 |
| Pipeline 生命周期管理 | ⚠️ 部分实现 | ✅ 已实现 | `api.py` 实现了完整的生命周期：create → trigger → pause → resume → terminate，带状态守卫（409 冲突检测） |

**关键改善**：
- 新增 `pipeline_config.py` 模块，提供声明式 DAG 依赖管理
- 新增 `api.py` REST API 层，暴露 pause/resume/terminate 操作
- `graph_runner.py` 增加生命周期守卫 `_assert_executable`

**仍存差距**：
- 自定义阶段名和自定义 Agent 仍被 `BUILT_IN_STAGE_AGENTS` 白名单拒绝
- DAG 依赖是声明式的，但实际执行仍通过函数调用链线性推进，不支持并行阶段执行

### 2. Agent 编排与执行

| 子需求 | 旧状态 | 新状态 | 说明 |
|---|---|---|---|
| Agent 角色定义（System Prompt） | ✅ 已实现 | ✅ 已实现 | 6 个 Agent 均有独立 prompt.py |
| 输入输出契约 | ✅ 已实现 | ✅ 已实现 | 所有 Agent 有 schema_version 契约 |
| Agent 感知代码库上下文 | ✅ 已实现 | ✅ 已实现 | workspace + 工具集（read/write/edit/glob/grep/powershell） |
| LLM Provider 可配置（至少2个） | ✅ 已实现 | ✅ 已实现 | 5 个 provider：ark、bailian、deepseek、openai、custom |
| 运行时可切换 | ⚠️ 部分实现 | ✅ 已实现 | 三层覆盖机制：API 参数 > 环境变量 > 配置文件。创建 Pipeline 时可通过 `provider` 字段覆盖，覆盖值持久化到 run payload |

**关键改善**：
- 运行时 Provider 切换从"仅配置文件级别"升级为三层覆盖机制
- API 创建 Pipeline 时支持 `provider` 字段，覆盖值会传递到 Graph Runner 执行层
- `graph_runner.py` 的 `load_config_for_run` 使用 `dataclasses.replace` 生成覆盖后的 LlmConfig

## 总体评价

相比旧评估文档，当前代码库在 **平台架构层面** 取得了重大进展：

1. **RESTful API 层从零到完整** — 10 个端点、OpenAPI 3.0.3 文档、20 个自动化测试，这是最大的改善
2. **Pipeline 生命周期管理补齐** — pause/resume/terminate 全部实现，带状态守卫
3. **运行时 LLM Provider 切换补齐** — 三层覆盖机制，API 级别可指定 provider
4. **检查点交互渠道扩展** — 从仅飞书卡片扩展为 REST API + 飞书三方审批 + CLI 三通道
5. **Pipeline 配置能力提升** — 支持自定义阶段列表（重排/删减）、DAG 依赖声明和循环检测

**Must-have 完成度**：5 项 Must-have 中，3 项完全满足（#2 Agent 编排、#3 HITL 检查点、#5 端到端演示），2 项基本满足但有细节差距（#1 Pipeline 引擎的可配置性/多Agent/并行、#4 API-First 的响应 Schema 定义不够完整）。

**Good-to-have 完成度**：6 项中 0 项完全实现，2 项有极简实现（自动回归、Pipeline 模板），4 项完全缺失。
